# Contributing to Elyan

## Principles

- Keep changes small and production-oriented.
- Prefer clear code over clever code.
- Preserve the local-first and evidence-first model.

## Workflow

1. Create a branch from `main`.
2. Make the change.
3. Run targeted tests.
4. Update docs if the surface changes.
5. Open a PR with a short summary and verification notes.

## Branch Naming

- `feature/<topic>`
- `fix/<topic>`
- `docs/<topic>`

## Pull Request Checklist

- [ ] Code is clear and minimal
- [ ] Tests pass for the changed surface
- [ ] Public docs are updated if needed
- [ ] Breaking changes are called out

## Commit Messages

Use short imperative messages, for example:

- `feat: add trace viewer`
- `fix: harden evidence resolver`
- `docs: publish public README`
