Getting Started

Start local. Keep private data local.

Elyan is a local-first personal agent runtime. The website exists to explain the product, guide installation, and provide the hosted account surface for credits, billing, and entitlements.

The default assumption is simple: private context stays on the user's machine. Hosted features are narrow and explicit.
Install

Use the real install path.

Elyan is installable on the user's own machine. The hosted website never becomes the default runtime.

brew install --formula https://raw.githubusercontent.com/emrek0ca/elyan/main/Formula/elyan.rb npm install -g github:emrek0ca/elyan elyan start --port 18789 elyan desktop
Local Setup

Ollama-first by default.

  • Run a local model lane with Ollama when possible.
  • Keep local files, local runtime state, and local actions on-device.
  • Cloud assistance is optional and explicit, never assumed.
brew install ollama ollama serve ollama pull llama3.2:3b
Optional Search

Search is optional, not the product.

Search can be enabled when the user explicitly wants it. It supports the local runtime, but does not replace it.

  • Use search only when needed.
  • Do not centralize personal memory into search by default.
  • Keep configuration narrow and reversible.
Optional MCP

Use MCP for explicit integrations.

Elyan can connect to narrow integrations through MCP when they are part of the user's workflow. It is an extension path, not the core boundary.

  • Integrations are opt-in.
  • Shared control-plane state does not equal private context.
  • Keep connectors and permissions explicit.
Dashboard + CLI

Desktop for the runtime. CLI for operators.

The local runtime is primarily a desktop product, with the CLI and dashboard surfaces used to inspect status, install, and operate the system.

CLI

Use the CLI for install, startup, health checks, and local operator workflows.

Dashboard

Use the panel for hosted account state, not as a fake local memory brain.

Hosted Account / Billing

elyan.dev is the account surface.

The hosted panel shows account, plan, entitlements, credits, usage, billing state, and notifications. It does not pretend to know private machine state unless that state is explicitly synced.

  • Register and login on elyan.dev.
  • See hosted credits and usage history.
  • Open billing or portal actions when the integration is actually available.
  • View product notices and account messages from the control plane.
Privacy / Boundaries

The boundary is a product requirement.

Local

Private memory, files, local tools, local actions, and user-private runtime state.

Hosted

Accounts, billing, entitlements, credits, usage, notifications, and install metadata.

Hosted state is narrow. It exists to support the product, not to absorb the user's private brain.

FAQ

Short answers.

Is Elyan a SaaS brain?

No. The local runtime is the product. Hosted is only the shared layer for account and billing state.

Does private context sync to the cloud by default?

No. Private context stays local unless the user explicitly enables a narrow sync path.

Where do I manage billing?

In the hosted panel after login. If billing integration is unavailable, the UI says so plainly.
