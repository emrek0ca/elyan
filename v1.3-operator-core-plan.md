# Elyan v1.3 Operator Core Plan

v1.3 is the Operator Core release.

The goal is to move Elyan from a stable local-first agent runtime into a real operator surface: sourced research, project cowork, coding workflows, and safe local automation. This is not a rewrite. The v1.2 runtime, CLI, local-agent, MCP/skills, hosted control-plane, release API, and VPS deployment model stay intact.

## Product Direction

Elyan v1.3 should feel closer to sourced research, project cowork, and coding-agent workflows while staying Elyan:

- local-first by default
- policy-first for every side effect
- typed actions before execution
- approval gates for risky work
- audit and evidence for operator actions
- honest unavailable states instead of fake capability claims

The core execution rule is:

```text
typed action -> policy -> approval -> execution -> audit -> verification
```

## Release Pillars

### External Inspiration Notes

v1.3 should learn from strong open-source agent/design projects without copying their code or importing their assumptions blindly.

- OpenMythos: adopt the product idea of adaptive depth. Elyan should not use one fixed reasoning path for every task. Operator runs should choose shallow, standard, or deep loops based on risk, uncertainty, and verification need.
- Taste Skill: adopt parameterized taste. Design work should expose density, motion, and variance as explicit decisions, not accidental model style.
- Huashu Design: adopt artifact-first design workflow. Design output should start from context and assets, produce inspectable prototypes/briefs, and include a review pass before delivery.

These inspirations must stay inside Elyan's policy model. They do not bypass typed actions, approval, audit, or verification.

### JS Library Strategy

Do not rebuild mature capability primitives from scratch. Prefer small, proven JavaScript libraries when they reduce risk and fit Elyan's local-first runtime.

Initial library direction:

- Repo/code intelligence: prefer `ts-morph`, `diff`, `simple-git`, `ignore`, and `fast-glob` style primitives before custom parsers.
- Terminal/process execution: prefer a controlled adapter around `execa` semantics rather than raw shell strings.
- Research and crawling: continue building on `crawlee`, `cheerio`, Playwright, and the existing citation/rerank path.
- Documents and artifacts: continue building on `docx`, `exceljs`, `pdf-lib`, `unpdf`, `mammoth`, `papaparse`, and `yaml`.
- UI/motion: continue using `framer-motion` and `lucide-react`; add heavier animation libraries only in isolated components when the product surface justifies it.
- Validation/contracts: continue using `zod` for every external input, action, run, approval, skill, and capability shape.

New dependencies require a concrete capability implementation and tests. Do not add libraries just because they are popular.

Implementation status:

- Active and planned JS primitives are exposed through the capability directory as a library strategy snapshot.
- Active libraries are already installed and used by existing capabilities.
- Planned libraries are documented as intended integration points only; they are not presented as available runtime capability until a concrete implementation and tests land.
- Code/process primitives remain inspect-and-plan-first. File mutation, command execution, dependency changes, git operations, and deploy actions must stay behind local-agent policy and approval.

### Operator Run Engine

Add a durable run model around the existing orchestration, team runtime, local-agent, MCP, and capability systems.

Each run must track:

- run id
- source and user intent
- selected mode
- plan steps
- approvals
- artifacts
- status
- verification result
- adaptive reasoning profile: depth, max passes, halting rule, stability guard, and rationale
- mode-specific quality gates: research source coverage, code approval/verification, cowork artifacts and memory boundary

### Research Mode

Make research a first-class mode:

- query expansion
- source collection
- citation enforcement
- duplicate cleanup
- source quality notes
- honest fallback when live search is unavailable

### Coding Mode

Make repository work a first-class mode:

- inspect repo state
- find relevant files
- plan edits
- propose patches
- run checks when approved
- summarize risks and verification

Mutating file, terminal, dependency, git, and deploy actions must go through local-agent policy.

### Cowork Project Mode

Make long-running work resumable:

- planner, researcher, executor, reviewer, verifier, memory curator roles
- artifacts for plan, research, execution, review, verification, and next steps
- local-first project memory
- no private local brain data sent to the hosted control-plane by default

### Safe Automation Layer

Extend local automation without weakening safety:

- filesystem actions
- terminal actions
- browser read/automation preview
- artifact creation
- run-linked evidence

Approval defaults:

- read-only: auto
- write-safe: confirm
- protected/sensitive: screen
- destructive/system-critical: two-factor or hard deny

### MCP and Skills Capability System

MCP tools and skills must appear as normalized capabilities:

- capability id
- input/output contract
- risk level
- approval requirement
- health state
- enabled/disabled state
- last error

MCP and skills must never bypass local policy, approval, audit, or verification.

### Command Center UX

`/manage` becomes the command center:

- active runs
- pending approvals
- connected device
- runtime health
- capabilities/MCP/skills status
- recent artifacts

## Public Interfaces

Additive CLI:

- `elyan run`
- `elyan runs list`
- `elyan runs show <runId>`
- `elyan approvals list`
- `elyan approvals approve <approvalId>`
- `elyan approvals reject <approvalId>`

Additive API:

- `GET /api/runs`
- `POST /api/runs`
- `GET /api/runs/[runId]`
- `GET /api/approvals`
- `POST /api/approvals/[approvalId]/approve`
- `POST /api/approvals/[approvalId]/reject`

## Definition of Done

v1.3 is release-ready only when:

- `npm run lint` passes
- `npm run test` passes
- `npm run build` passes
- `npm run release:check` passes
- research mode returns sourced answers or honest fallback
- coding mode can inspect, plan, and verify without bypassing approvals
- risky local actions require approval
- run artifacts are inspectable through API and command center
- MCP/skills states are visible and policy-bound

## Current v1.3 Implementation Notes

- Operator runs now persist adaptive reasoning profiles and mode-specific quality gates.
- Research runs pass only when they record sources/citations or an honest unavailable-state artifact.
- Code runs remain blocked until approval, repository inspection evidence, and verification evidence are recorded.
- Cowork runs track role artifacts and local memory boundary gates.
- `/manage` and CLI list quality gate progress without executing side effects.
